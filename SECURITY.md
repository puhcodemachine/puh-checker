# Безопасность ПУХ-checker

Документ для боевого деплоя на свой сервер + домен. Речь о средствах клиентов — относиться серьёзно.

## Что уже сделано в коде (серверная версия `server.py`)

- **Шифрование хранилища на диске.** `data/tasks.json` хранится зашифрованным (Fernet/AES-128-CBC+HMAC). Ключ — в файле `.enc_key` (права `600`, в `.gitignore`, в репозиторий не попадает). Сиды на диске в открытом виде НЕ лежат. Старый открытый JSON мигрируется автоматически при первой записи.
- **Настоящая серверная авторизация.** Логин/пароль; пароль хранится как PBKDF2-SHA256 хэш в `auth.json` (`600`, gitignore), сравнение constant-time (`hmac.compare_digest`).
- **Сессии с истечением** (12 ч), id 32 байта (`secrets`), cookie `HttpOnly` + `SameSite=Strict` + `Secure` (на проде).
- **Анти-брутфорс**: блок входа после 8 неудач за 10 мин на IP.
- **Заголовки безопасности** на каждый ответ: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`, `X-Robots-Tag: noindex`, `Permissions-Policy`.
- **Лимит тела запроса** (анти-DoS), версия сервера скрыта.
- **XSS**: весь пользовательский ввод экранируется (`esc`, включая кавычки) перед вставкой в DOM.
- **Права файлов**: `data/` → `700`, `.enc_key`/`auth.json`/`tasks.json` → `600`.

## Что ТЫ обязан сделать при деплое

1. **HTTPS.** Только по HTTPS. Домен → сертификат Let's Encrypt (`certbot`) или Cloudflare (Full/Strict + Origin-сертификат). Выстави переменную окружения **`PUH_HTTPS=1`** — тогда cookie станет `Secure` (не уйдёт по HTTP).
2. **Смени пароль.** Текущий светился в переписке. Сгенерируй новый хэш:
   ```
   python3 -c "import hashlib,secrets,json;s=secrets.token_bytes(16);pw=input('пароль: ');open('auth.json','w').write(json.dumps({'user':'Puh','salt':s.hex(),'iters':200000,'hash':hashlib.pbkdf2_hmac('sha256',pw.encode(),s,200000).hex()}))"
   chmod 600 auth.json
   ```
3. **Бэкап ключа `.enc_key`** в надёжном месте ОТДЕЛЬНО от сервера. Потеряешь ключ — потеряешь доступ к зашифрованным заданиям. Утечёт ключ + диск — расшифруют. Храни ключ и бэкап данных раздельно.
4. **Фаервол**: наружу открыт только 443 (и 80 для редиректа на 443). Порт приложения (8777) — только локально (за nginx), не наружу. SSH — по ключу, не по паролю.
5. **Запуск под отдельным пользователем** (не root), systemd с `NoNewPrivileges`, `ProtectSystem=strict`, `ProtectHome`, доступ на запись только к `data/`.
6. **Бэкапы** `data/` и `auth.json` — только в зашифрованном виде.
7. **Логи** не должны писать сиды/пароли (сейчас логи приглушены).

## Остаточные риски (честно)

- **Полная компрометация сервера (RCE/root)** = доступ к ключу `.enc_key` в памяти процесса → расшифровка. Шифрование на диске защищает от утечки диска/бэкапа/репозитория, но НЕ от взломанного работающего сервера. Защита: минимум поверхности, обновления, изоляция, мониторинг.
- **Балансы** проверяются через публичные API — провайдеры видят адреса (не сиды). Для приватности — свои ноды.
- **Максимальная защита** (zero-knowledge: сервер вообще не видит сиды в открытом виде) требует шифрования на стороне клиента ключом из пароля — это отдельный большой этап, можем сделать.

## Публичная демо-витрина (GitHub Pages) — НЕ для боевых данных

`puhcodemachine.github.io` — это статичная демо-версия: задачи хранятся в `localStorage` браузера **открытым текстом**, «барьер»-пароль косметический, домен общий на все репозитории аккаунта. **Реальные клиентские сиды туда вводить нельзя.** Боевая работа — только на серверной версии (HTTPS + шифрование + логин).
